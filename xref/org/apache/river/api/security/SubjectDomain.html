19  package org.apache.river.api.security;
20  
21  import java.security.ProtectionDomain;
22  import javax.security.auth.Subject;
23  
24  /**
25   * A Marker interface used by <code>net.jini.security.Security</code> to mark
26   * a {@link java.security.ProtectionDomain} added to the stack that represents an
27   * authenticated {@link javax.security.auth.Subject} with 
28   * {@link java.security.Principal}'s.
29   * <p>
30   * Typically in non-distributed systems, a <code>Subject</code> is represented
31   * by a <code>Thread</code> and all {@link java.security.Permission} granted 
32   * to that <code>Subject</code> is injected into every <code>ProtectionDomain</code> 
33   * present on the call stack. Code is always trusted, or at least it must be at the time the 
34   * <code>Subject</code> is authenticated.
35   * <p>
36   * However in a distributed system, because code trust and Subject's are 
37   * separate concerns, methods have been provided in 
38   * <code>net.jini.security.Security</code> to push a <code>SubjectDomain</code>
39   * onto the stack instead of injecting Principal's into all ProtectionDomain's on 
40   * the stack, this avoids mistakenly elevating privileges of less trusted
41   * code, if present on a call stack at the time the doAs subject method is called.
42   * <p>
43   * When using these methods, the security Policy needs to be written to grant
44   * Permission to code signer certificates and Principals separately.
45   * <p>
46   * For example, a RemotePolicy service is updated by an Administrator client
47   * using PermissionGrant's, however one of these grants is not signed by a 
48   * trusted Certificate.  Ordinarily this PermissionGrant would run with the
49   * privileges of the administrator, but because code trust and user trust
50   * should be considered separate concerns in a distributed system,
51   * the administrator is unable to make these PermissionGrant's in the 
52   * presence of untrusted code.
53   * <p>
54   * These methods apply the principle of least privilege to Subject's as well
55   * as code.  The privileges allowed will be the intersection of Permission 
56   * granted to each ProtectionDomain on the call stack.
57   * <p>
58   * These methods require a River or Jini Policy provider to be installed.  The
59   * SubjectDomain is a dynamic ProtectionDomain, it contains no Permission,
60   * it always consults the current {@link java.security.Policy}.
61   * 
62   * <br><code>see net.jini.security.Security#doAs(javax.security.auth.Subject, java.security.PrivilegedAction)</code>
63   * <br><code>see net.jini.security.Security#doAs(javax.security.auth.Subject, java.security.PrivilegedExceptionAction)</code>
64   * <br><code>see net.jini.security.Security#doAsPrivileged(javax.security.auth.Subject, java.security.PrivilegedAction, net.jini.security.SecurityContext) </code>
65   * <br><code>see net.jini.security.Security#doAsPrivileged(javax.security.auth.Subject, java.security.PrivilegedExceptionAction, net.jini.security.SecurityContext)</code> 
66   * @see net.jini.security.policy.DynamicPolicyProvider
67   * @see org.apache.river.api.security.ConcurrentPolicyFile
68   * 
69   * @author Peter Firmstone
70   * @since 3.0.0
71   */
72  public interface SubjectDomain {
73      
74      public Subject getSubject();
75  }
