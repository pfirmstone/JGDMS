19  package org.apache.river.api.security;
20  
21  import java.io.IOException;
22  import net.jini.security.GrantPermission;
23  import net.jini.security.policy.UmbrellaGrantPermission;
24  import org.apache.river.api.common.Beta;
25  
26  /**
27   * <p>
28   * RemotePolicy is a service api that can be implemented by a distributed Policy service, 
29   * allowing local Policy providers to be updated remotely by a djinn group administrator.
30   * </p><p>
31   * No service implementation has been provided, RemotePolicyProvider
32   * implements this interface to simplify creation of such a service.
33   * </p>
34   * <h2>Notes for implementors:</h2>
35   * <p>
36   * For security purposes, only secure jeri Endpoint's should be used and must
37   * require client and server authentication, in addition the proxy must be a 
38   * reflective proxy only, as DownloadPermission should not be granted, which is 
39   * also beneficial to reduce network load on the administrator client.  
40   * RemotePolicy may be submitted to a lookup service, where an administrator 
41   * client will respond to a ServiceEvent notification, thereby providing the
42   * RemotePolicy service node, with the permissions requried to participate in 
43   * the djinn, an administrator client will also periodically update policy in
44   * the djinn.
45   * </p><p>
46   * To reduce network load, the administrator client may delay updates by
47   * lazily processing updates in a serial manner.  New RemotePolicy services
48   * obtained by the administrator client's via RemoteEvent notification should
49   * be processed as a priority over policy updates.  Eventually a djinn group
50   * policy should reach equilibrium where all nodes have had their policy's 
51   * updated.
52   * </p><p>
53   * This policy, in addition to any local policy provider, allows a network djinn
54   * administrator to provide a list of PermissionGrant's, from a single or 
55   * replicated remote location,  distributed to all nodes in a djinn.
56   * </p><p>
57   * Multiple RemotePolicy's may be in force on a single JVM instance,
58   * each nested policy service instance may be responsible for updating policy for each
59   * djinn or group a JVM node joins.
60   * </p><p>
61   * RemotePolicy provides a means to dynamically grant permissions required 
62   * by other services provided by the djinn or client Subjects accessing
63   * services provided by the djinn. 
64   * The JVM that receives policy in the form of PermissionGrant's should limit
65   * the GrantPermission's granted to a djinn administrator, especially if that
66   * administrator is from another organisation.
67   * </p><p>
68   * In addition, replicating administrator clients may register a pseudo RemotePolicy
69   * in order to track the primary administrator client and take over in the
70   * event it fails.  Failure may be failure to authenticate or failure to renew
71   * a Lease.
72   * </p><p>
73   * RemotePolicy, if it encapsulates an underlying RemotePolicy, does not
74   * delegate updates to the base RemotePolicy, this is in case an
75   * implementer wants a number of different layers of RemotePolicy, where
76   * each layer represents a different administrator role or responsibility.  
77   * The administrator's subject must hold the necessary permissions in order
78   * to grant them, including GrantPermission and PolicyPermission("REMOTE").
79   * </p><p>
80   * A node may join more than one djinn group, in this case RemotePolicy's may
81   * be used as nested basePolicy's.
82   * </p><p>
83   * The intent of RemotePolicy is for granting of DowloadPermission to
84   * new signer Certificates and adding new Principals and Permission's to
85   * distributed policy providers.
86   * </p><p>
87   * Local policy files should be used to restrict the Permissions grantable
88   * via a RemotePolicy.
89   * </p><p>
90   * PermissionGrant's that are replaced and no longer exist in the RemotePolicy
91   * will no longer be implied by the policy.
92   * </p><p>
93   * DefaultPolicyParser has been provided for an administrator client to
94   * parse standard java format policy file's, to create PermissionGrant's.
95   * </p>
96   * @author Peter Firmstone
97   * @since 3.0.0
98   * @see GrantPermission
99   * @see UmbrellaGrantPermission
100  * @see PolicyParser
101  * @see PolicyPermission
102  */
103 @Beta
104 public interface RemotePolicy {
105     /**
106      * Replaces the existing RemotePolicy's PermissionGrant's.
107      * 
108      * The array is defensively copied, the caller, must have 
109      * RuntimePermission("getProtectionDomain")
110      * as well as GrantPermission or UmbrellaGrantPermission for every
111      * Permission granted by each PermissionGrant.
112      * 
113      * If the calling Subject doesn't have sufficient permission, the 
114      * first permission that fails will include the SecurityException as the
115      * cause of the thrown IOException.
116      * 
117      * Permissions required by the callers Subject should be set in the 
118      * local policy files at the RemotePolicy server.
119      * 
120      * When granting a Permission, if its class is contained
121      * within a downloaded code, remember to use UnresolvedPermission instead,
122      * in case the class isn't present at the client.
123      * 
124      * Where an IOException is thrown, no update to the
125      * RemotePolicy has occurred.
126      * 
127      * @param policyPermissions PermissionGrant array.
128      * @throws java.io.IOException if connection problem occurs.
129      */
130     public void replace(PermissionGrant[] policyPermissions) throws IOException;
131 }
