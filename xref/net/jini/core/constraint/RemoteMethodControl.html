19  package net.jini.core.constraint;
20  
21  /**
22   * Defines an interface to proxies for setting constraints to control remote
23   * method calls. If a proxy implements this interface, then the semantics of
24   * this interface apply to all calls through all remote methods defined by the
25   * proxy; there is no mechanism to exempt remote methods or remote interfaces
26   * from these semantics.
27   * <p>
28   * Constraints for a remote call come from two sources:
29   * <ul>
30   * <li>Constraints imposed by the server (including any minimum constraints
31   * imposed by the communication mechanism used between the proxy and server)
32   * <li>Constraints placed on a proxy by the client
33   * </ul>
34   * The server constraints are controlled by the proxy implementation; they are
35   * not exposed to the client, and might vary in ways unknown to the client
36   * (for example, vary by method or over time). The client should set the
37   * constraints it wants rather than assuming that the server imposes
38   * particular constraints. Client constraints placed on a proxy apply to all
39   * remote calls made through that particular proxy by any thread, and the
40   * client can specify different constraints for each remote method.
41   * <p>
42   * A remote call will be performed only if the combined requirements (from
43   * both sources) can be satisfied. If the combined requirements cannot be
44   * satisfied, a {@link java.rmi.ConnectIOException} will be thrown by the
45   * remote call, typically containing (but not required to contain) a nested
46   * {@link net.jini.io.UnsupportedConstraintException}.
47   * In addition to the requirements, both client and server preferences will
48   * be satisfied, to the extent possible.
49   * <p>
50   * Note that constraints imposed by the communication mechanism must be
51   * factored into the requirements. For example, if the only explicit
52   * requirement is <code>Delegation.YES</code>, but the communication
53   * mechanism always requires client authentication, then effectively a
54   * <code>ClientAuthentication.YES</code> requirement exists, and so the
55   * <code>Delegation.YES</code> requirement must also be satisfied.
56   * <p>
57   * The constraint mechanisms are designed such that client constraints do not
58   * weaken server constraints, and vice versa. However, it is certainly
59   * possible to specify conflicting constraints. Preferences that conflict
60   * with requirements are ignored, and if preferences conflict with each other
61   * it is arbitrary as to which (if any) are satisfied, but if there are
62   * conflicting requirements the remote call will not be made.
63   *
64   * @author Sun Microsystems, Inc.
65   * @since 2.0
66   */
67  public interface RemoteMethodControl {
68      /**
69       * Returns a new copy of this proxy with the client constraints set to the
70       * specified constraints. These constraints completely replace (in the
71       * copy) any client constraints previously placed on this proxy; calling
72       * the {@link #getConstraints getConstraints} method of the copy returns
73       * the identical constraints instance. The original proxy is not modified.
74       * A <code>null</code> value is interpreted as mapping all methods to
75       * empty constraints (one that has no requirements and no preferences).
76       * For any given remote call, the specific client requirements and
77       * preferences to be satisfied are given by the return value of invoking
78       * the {@link MethodConstraints#getConstraints getConstraints} method of
79       * the specified {@link MethodConstraints} instance with a
80       * {@link java.lang.reflect.Method} object representing the remote method.
81       * <p>
82       * Client constraints placed on a proxy are included in the serialized
83       * state of the proxy. This allows third-party services to be transparent
84       * to the client's needs. For example, if remote object <code>s1</code>
85       * obtains a proxy for remote object <code>s2</code>, and passes that
86       * proxy to remote object <code>s3</code>, expecting <code>s3</code> to
87       * invoke a remote method on <code>s2</code>, then <code>s1</code> can
88       * control that call by placing its constraints directly on the proxy
89       * before passing it to <code>s3</code>. If <code>s3</code> does not
90       * wish to be transparent in this way, then it should explicitly replace
91       * the client constraints on received proxies with whatever constraints
92       * are appropriate to implement its own policy.
93       *
94       * @param constraints client constraints, or <code>null</code>
95       * @return a new copy of this proxy with the client constraints set to the
96       * specified constraints
97       * @see #getConstraints
98       */
99      RemoteMethodControl setConstraints(MethodConstraints constraints);
100 
101     /**
102      * Returns the client constraints placed on this proxy. The return
103      * value can be <code>null</code>, which is interpreted as mapping all
104      * methods to empty constraints (one that has no requirements and no
105      * preferences).
106      *
107      * @return the client constraints, or <code>null</code>
108      * @see #setConstraints
109      */
110     MethodConstraints getConstraints();
111 }
